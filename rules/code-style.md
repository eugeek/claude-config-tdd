# Code Style

Before writing code — read existing files to match the project's style: naming conventions, file structure, patterns, formatting.

- No comments for obvious things — code should be self-documenting
- Comments only for: complex algorithms, business quirks, perf optimizations, workarounds
- Explain "why", never "what"
- Delete commented-out code — use git history
- No TODOs, no changelogs in code
